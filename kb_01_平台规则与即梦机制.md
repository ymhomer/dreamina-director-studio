# KB-01｜平台规则与即梦机制

> 用途：本知识库用于帮助「即梦导演 Prompt Studio」在生成、优化、拆解、续写和教学时，正确理解即梦 / Dreamina / Seedance 类 AI 视频平台的基础机制、输入输出限制、@引用逻辑、数字人规则、字幕文字策略、合规边界与常见失败原因。

> 调用优先级：当用户的问题涉及平台功能、视频时长、比例、@分身、图生视频、文生视频、参考图、参考视频、字幕、文字、拍同款、数字人、人物一致性、生成失败原因、平台限制时，应优先调用本库。

## 1. 知识库定位

本库不是创意库，也不是风格库，而是平台机制与生成规则库。

它的核心作用是帮助 GPT 判断：

1. 用户的需求是否适合即梦生成。
2. 用户的提示词是否符合平台生成逻辑。
3. 是否需要使用 @引用、参考图、首帧、尾帧或数字人。
4. 是否存在时长、比例、字幕、人物、版权或复杂度风险。
5. 是否需要把用户需求改写成更稳定、更符合平台理解方式的自然语言 Prompt。

在完整 GPT 工作流中，本库主要负责「可执行性检查」和「平台适配检查」。

```mermaid
flowchart LR
    A[用户需求] --> B[判断平台适配]
    B --> C[检查输入素材]
    C --> D[检查时长比例]
    D --> E[检查@引用与数字人]
    E --> F[检查字幕文字]
    F --> G[检查合规风险]
    G --> H[输出更稳定的即梦Prompt]
```

## 2. 即梦 / Dreamina / Seedance 的基础理解

即梦类 AI 视频平台的核心不是传统剪辑软件，而是「多模态视频生成系统」。

它通常支持以下输入：

* 文字 Prompt
* 图片参考
* 视频参考
* 音频参考
* 首帧 / 尾帧
* 数字人 / AI 分身
* @引用素材

它的输出通常是短视频，因此生成逻辑更接近：

```text
用户意图 + 视觉参考 + 文字描述 + 镜头语言 + 风格约束 + 平台模型理解 = AI生成视频
```

即梦 Prompt 的关键不是写得越长越好，而是让模型清楚理解：

```text
谁，在什么场景，做什么动作，镜头怎么拍，画面是什么风格，最后停在哪里。
```

## 3. 默认平台参数建议

如果用户没有特别说明，GPT 应使用以下默认值：

| 项目       | 默认建议                                               |
| -------- | -------------------------------------------------- |
| 视频比例     | 9:16 竖屏                                            |
| 视频时长     | 12 秒                                               |
| 视频类型     | 短视频 / 抖音式节奏                                        |
| 主体数量     | 1–2 个主体最稳                                          |
| 镜头数量     | 3–4 个节奏段                                           |
| 结构       | 前 2–3 秒钩子 + 中段推进 + 结尾记忆点                           |
| 字幕       | 建议后期添加，除非用户明确要求画面内生成                               |
| 风格       | 只保留 1 个主风格，可加 1 个辅助质感                              |
| 输出格式     | 优先 Canvas，方便修改；Markdown code block 作为后补，方便复制       |
| 分隔线      | 不使用 `---`                                          |
| Prompt长度 | 先输出完整导演版，不因字数提前删减；再额外输出 ≤580字压缩版，包含符号、空格与换行，并标注字符数 |

## 4. 文生视频、参考图与角色调用逻辑

本节用于定义即梦 Prompt 中最重要的三类输入逻辑：

1. 文本生成逻辑：用户只靠文字描述生成视频。
2. 参考图逻辑：用户提供图片作为视觉参考。
3. 角色调用逻辑：用户使用 @角色 / @分身 指定人物身份。

三者可以单独使用，也可以组合使用，但职责必须分清。

```mermaid
flowchart TD
    A[用户输入] --> B{输入类型判断}
    B -->|只有文字| C[文生视频逻辑]
    B -->|有参考图| D[参考图逻辑]
    B -->|有多张参考图| D2[多参考图逻辑]
    B -->|图片素材需求| D3[素材拆分逻辑]
    B -->|有@角色/@分身| E[角色调用逻辑]
    D --> F[判断参考图用途]
    D2 --> F2[逐张判断参考图用途]
    E --> G[锁定人物身份]
    F --> H[组装Prompt]
    F2 --> H
    D3 --> H
    G --> H
    C --> H
```

## 5. 文生视频逻辑

文生视频适合以下情况：

* 用户只有概念，没有参考图。
* 生成原创场景、原创角色、原创风格。
* 主题偏抽象、幻想、概念化。
* 不要求严格复刻某个人物外貌。

文生视频 Prompt 应完整描述：

```text
主体 + 场景 + 动作 + 镜头 + 光色 + 风格 + 声音 + 约束
```

适合示例：

```text
一个古代侠客站在雨夜客栈门口，镜头从低角度缓慢推进，暖黄灯笼光与冷蓝雨夜形成对比，侠客抬头露出坚定眼神，远处传来鼓点和风声，整体为90年代港片武侠胶片质感。
```

文生视频的重点是「描述完整」，因为没有外部视觉参考帮助模型理解。

## 6. 参考图逻辑

### 6.1 参考图的本质

参考图是视觉参考，不是角色身份本身。

参考图通常用于帮助模型理解：

* 画面构图
* 人物造型
* 服装妆造
* 场景空间
* 光影方向
* 色彩氛围
* 产品外观
* 姿势动作
* 首帧画面
* 整体视觉风格

参考图的核心写法是：

```text
参考图作为{具体用途}参考
```

### 6.2 参考图用途分类

当用户提供参考图时，GPT 必须先判断参考图的主要用途，再写入 Prompt。

| 参考图用途  | Prompt 写法              |
| ------ | ---------------------- |
| 人物造型参考 | 参考图作为人物造型、服装、发型和整体气质参考 |
| 首帧构图参考 | 参考图作为首帧构图，镜头从该画面开始     |
| 场景参考   | 参考图作为场景空间、背景元素和环境氛围参考  |
| 光影参考   | 参考图作为光线方向、明暗关系和色彩氛围参考  |
| 风格参考   | 参考图作为整体视觉风格、质感和调色参考    |
| 产品参考   | 参考图中的产品作为核心物件，外观保持清晰稳定 |
| 姿势参考   | 参考图中的姿势作为起始动作，后续动作自然延展 |
| 尾帧参考   | 结尾停留在接近参考图的构图和情绪状态     |

### 6.3 参考图 Prompt 稳定写法

低效写法：

```text
参考这张图，拍一个很高级的视频。
```

稳定写法：

```text
参考图作为主角造型与服装参考，保持发型、服装颜色和整体气质一致。画面从中景开始，主角缓慢转身看向镜头，背景为霓虹雨夜街道，镜头稳定慢推，整体港风电影质感。
```

首帧写法：

```text
参考图作为首帧构图，画面从参考图的构图开始，主角保持原本姿势，随后缓慢抬头看向镜头，镜头轻微推进，光影和色调延续参考图。
```

风格写法：

```text
参考图作为整体视觉风格参考，保留其光影方向、色彩氛围和画面质感，但主体动作和剧情重新生成。
```

产品写法：

```text
参考图中的产品作为核心物件，保持外观、颜色和形状清晰稳定，镜头从产品特写缓慢拉开，展示使用场景。
```

### 6.4 多张参考图使用逻辑

当用户提供多张参考图时，GPT 必须把它们视为不同的视觉任务来源，而不是把所有参考图混成一个模糊要求。

即梦提示词中应沿用用户素材命名，例如：

```text
参考图
参考图2
参考图3
参考图4
```

多张参考图的核心写法是：

```text
参考图作为{用途A}参考；参考图2作为{用途B}参考；参考图3作为{用途C}参考。
```

多张参考图可分别承担不同任务：

| 素材名称     | 常见用途                   |
| -------- | ---------------------- |
| 参考图      | 主视觉、首帧构图、主要场景、整体风格     |
| 参考图2     | 服装造型、发型、配件、人物姿势        |
| 参考图3     | 动作关系、站位构图、镜头节奏、尾帧情绪    |
| 参考图4 及以上 | 道具、产品、局部细节、补充光影或其他辅助参考 |

多参考图稳定写法：

```text
参考图作为主场景与光影氛围参考；参考图2作为男女主服装、发型和配件参考；参考图3作为双人站位、动作关系和镜头构图参考。@角色A与@角色B负责人物身份，人物面部以@角色为准，参考图只负责视觉参考。
```

如果用户明确说明每张图用途，应完全遵守用户指定。

示例：

```text
参考图作为背景设计参考，保留舞台空间、灯光方向和色彩氛围；参考图2作为男女主造型参考，保留服装层次、发型和配件；参考图3作为动作描绘草图参考，保留人物站位、互动方向和姿势关系。主角身份由@角色A与@角色B负责，脸部不要参考图片人物。
```

如果用户没有说明用途，GPT 应根据图片内容自动判断最合理的分工，优先采用以下顺序：

```text
场景 / 背景 → 服装 / 造型 → 动作 / 站位 → 光影 / 风格 → 道具 / 细节
```

多张参考图使用原则：

1. 每张参考图都要写清用途。
2. 不要把多张参考图都写成「整体参考」。
3. 不要让多张参考图同时决定同一个核心元素，除非用户明确要求融合。
4. 如果参考图之间冲突，应指定主参考图和辅助参考图。
5. 人物身份始终以 @角色 / 数字人为准，参考图只负责造型、服装、场景、动作或风格。
6. 若参考图中有人脸，但用户只要服装或动作，应明确写「参考图只作为服装/动作参考，不参考人脸身份」。
7. 若参考图用于即梦素材图拆解，可将其分别标注为「背景设计图」「造型图」「动作描绘草图」。
8. 多参考图 Prompt 不宜过长，应优先保留每张图的主要任务。

多参考图冲突处理：

| 冲突内容 | 处理方式                |
| ---- | ------------------- |
| 场景冲突 | 以用户指定的主场景参考图为准      |
| 服装冲突 | 以造型参考图为准            |
| 人脸冲突 | 以 @角色 / 数字人为准       |
| 动作冲突 | 以动作参考图或用户文字指令为准     |
| 风格冲突 | 以用户指定主风格为准，其次主视觉参考图 |
| 构图冲突 | 首帧参考优先，其次动作参考图      |

多参考图压缩写法：

```text
参考图负责场景与光影；参考图2负责服装造型；参考图3负责动作站位。人物身份以@角色为准，动作重新生成，镜头稳定，风格统一。
```

### 6.5 参考图使用原则

1. 先判断参考图用途，再写 Prompt。
2. 一张参考图最好只承担一个主要任务。
3. 如果参考图承担多个任务，必须写清主次。
4. 人物图可以参考造型和气质，但人物身份不等于固定分身。
5. 若用户要指定某个固定人物身份，应使用角色调用逻辑。
6. 参考图用于风格时，不应强行继承图中所有元素。
7. 参考图用于首帧时，动作应从稳定姿势开始。
8. 参考图用于产品时，产品外观优先于复杂运镜。
9. 多张参考图必须逐张说明用途，避免模型混淆。
10. 多张参考图之间存在冲突时，应明确主参考图与辅助参考图。

### 6.6 图片素材准备机制

当用户提出「图片素材」「素材参考图」「先处理素材」「拆分三张参考图」「生成即梦素材图」等需求时，GPT 应先进入素材准备模式，而不是直接生成最终视频 Prompt。

素材准备模式的输入可以是：

* 原始参考图
* 纯文字描述
* 图片生成提示词
* 视频生成提示词
* 图文混合需求
* 多张参考图 + 补充说明
* 已有概念 + 想延伸的视频内容

输出最多三张参考图需求：

| 素材图                 | 主要用途       | 核心要求                                                  |
| ------------------- | ---------- | ----------------------------------------------------- |
| 参考图1：纯背景图           | 场景、光影、空间氛围 | 无完整人物、无清晰完整人脸、无完整正面肖像，只保留场景、道具、光线、雾气、装饰和空间层次          |
| 参考图2：服装、发型、妆容与配饰展示图 | 造型、材质、妆容色彩 | 可用无脸人台、平铺展示、服装设计稿、发型背影、假发展示、分离式妆容板；妆容元素可分开出现，但不能组成完整脸 |
| 参考图3：动作与运镜草图        | 姿态、站位、镜头路径 | 使用分镜草图、动作设计图、无脸轮廓、火柴人、背影或剪影；可加箭头、推拉摇移标记、景别变化提示        |

灵活处理原则：

1. 三张图不是死板固定，需根据用户输入决定强化重点。
2. 场景最重要时强化背景图；造型最重要时强化造型图；视频动作最重要时强化动作与运镜草图。
3. 即使输入只有图片需求，也可推导适合视频延伸的动作与镜头素材。
4. 不生成可识别身份的人脸，不让眼睛、鼻子、嘴巴和脸型组合成完整五官。
5. 可出现分离式五官或妆容素材，例如单眼眼妆、唇色卡、眉形线稿、腮红位置图，但必须分散呈现。
6. 素材图只承担背景、造型、动作或镜头参考，不等同 @分身身份。

素材准备输出模板：

```text
【参考图1｜纯背景图】{场景风格、色调、光影、空间层次、道具与氛围，无完整人物/无清晰人脸}
【参考图2｜造型展示图】{服装、发型、配饰、材质、分离式妆容元素，使用无脸人台/平铺/设计板}
【参考图3｜动作与运镜草图】{无脸人物轮廓、站位、动作节奏、镜头路径、箭头、景别变化}
```

## 7. 角色调用逻辑

### 7.1 角色调用的本质

角色调用用于指定「谁是主角」。

当用户使用 @角色、@分身、@数字人时，GPT 应理解为用户要锁定某个已存在的人物身份。

常见写法：

```text
@角色A作为主角
```

```text
@角色B作为主角
```

```text
@角色C作为舞台中心主唱
```

角色调用主要负责：

* 人物身份
* 面部稳定
* 固定角色连续性
* 系列视频人物一致
* 双人或多人关系

### 7.2 单人角色调用写法

```text
@角色A作为主角，保持面部、发型与身份一致，穿简洁蓝色西装，站在TED演讲舞台中央，镜头从中景缓慢推进到近景，他自信看向观众，舞台灯光干净专业。
```

### 7.3 双人角色调用写法

```text
@角色A站在画面左侧，@角色B站在画面右侧，两人面对镜头轮流对嘴演唱，动作同步但不重叠，镜头稳定中景，保持两人面部清晰、服装一致、无多余人物。
```

### 7.4 多角色调用规则

多角色场景必须明确：

* 谁是主角
* 谁是配角
* 每个人的位置
* 每个人的动作
* 互动关系
* 镜头优先关注谁

推荐写法：

```text
@角色A为主角站在画面中央，@角色B在右侧微笑看向他，另一名路人只作为模糊背景，不抢主体。镜头优先跟随@角色A的动作。
```

### 7.5 角色稳定约束

当 Prompt 中出现 @角色时，应自动加入稳定约束：

```text
保持@角色面部一致，服装发型一致，主体始终清晰居中，动作连贯缓慢，避免面部变形，无多余人物，无穿模。
```

### 7.6 高风险动作与替代方案

以下动作容易导致角色不稳定：

* 快速翻滚
* 高速打斗
* 连续转身
* 大幅度跳舞
* 多人拥挤互动
* 镜头快速绕脸
* 遮挡脸部后快速切换
* 强特效覆盖面部

更稳替代：

| 高风险动作  | 稳定替代        |
| ------ | ----------- |
| 快速打斗   | 慢动作蓄力、定格出招  |
| 疯狂跳舞   | 简洁手势 + 节奏点头 |
| 快速转圈   | 半身缓慢转身      |
| 多人贴脸互动 | 双人中景，明确左右位置 |
| 强光覆盖脸  | 轮廓光，不遮挡五官   |

## 8. 参考图 + 角色组合逻辑

当用户同时提供参考图和 @角色时，必须明确分工：

```text
@角色负责人物身份。
参考图负责视觉参考。
```

### 8.1 组合写法：参考图用于服装

```text
@角色A作为主角，保持本人面部与身份一致；参考图作为服装和造型参考，保留参考图中的蓝色西装、干净发型和专业气质。镜头从中景缓慢推进，主角自信看向镜头。
```

### 8.2 组合写法：参考图用于场景

```text
@角色B作为主角，保持面部稳定；参考图作为场景与光影参考，保留参考图中的湖边、柔和自然光和白色蕾丝伞氛围。女主缓慢转身看向镜头，画面唯美稳定。
```

### 8.3 组合写法：参考图用于首帧

```text
@角色A作为主角，参考图作为首帧构图与服装风格参考，画面从参考图相似构图开始，主角缓慢抬头看向镜头，随后镜头稳定推进。
```

### 8.4 组合写法：参考图用于风格

```text
@角色B作为主角，保持人物身份一致；参考图作为整体视觉风格参考，延续其柔和光影、浅色调和写真质感，但动作和场景重新生成。
```

### 8.5 组合写法：多张参考图 + @角色

当用户同时提供多张参考图和 @角色时，GPT 必须同时处理「人物身份」与「素材分工」。

标准写法：

```text
@角色A与@角色B负责人物身份，保持两人面部、发型和身份一致；参考图作为主场景与光影参考，参考图2作为服装造型参考，参考图3作为动作站位参考。人物脸部以@角色为准，参考图不作为人脸身份来源。
```

如果参考图中有人脸，但用户只需要造型或动作：

```text
参考图2只作为服装、发型和配件参考，不参考图中人物脸部身份；主角面部以@角色A为准。
```

如果三张参考图分别是素材图：

```text
参考图作为背景设计图参考；参考图2作为造型图参考；参考图3作为动作描绘草图参考。@角色A与@角色B负责人物身份，画面生成时保持角色脸部稳定，动作和场景根据三张参考图分工组合。
```

多张参考图 + @角色优先级：

| 内容        | 优先级                |
| --------- | ------------------ |
| 人物身份 / 面部 | @角色优先              |
| 服装 / 妆造   | 用户指定优先，其次造型参考图     |
| 场景 / 光影   | 用户指定优先，其次场景参考图     |
| 动作 / 站位   | 用户文字指令优先，其次动作参考图   |
| 构图 / 首帧   | 首帧参考图优先            |
| 风格 / 调色   | 用户指定主风格优先，其次主视觉参考图 |

### 8.6 组合优先级

当参考图和 @角色信息冲突时，优先级如下：

| 冲突内容      | 优先级            |
| --------- | -------------- |
| 人物身份 / 面部 | @角色优先          |
| 服装 / 妆造   | 用户明确指定优先，其次参考图 |
| 场景 / 光影   | 参考图优先          |
| 动作 / 剧情   | 用户文字指令优先       |
| 风格 / 调色   | 用户指定风格优先，其次参考图 |

### 8.7 错误组合方式

错误：

```text
参考图中的人就是@角色A，完全变成参考图的人。
```

问题：人物身份和参考图人物混在一起，容易导致脸跑偏。

更稳：

```text
@角色A作为主角，保持本人面部与身份一致；参考图只作为服装、发型和光影风格参考。
```

错误：

```text
参考图作为人物、场景、动作、风格、首帧、尾帧全部参考。
```

问题：参考任务过多，模型容易混乱。

更稳：

```text
参考图主要作为首帧构图和光影风格参考，人物身份以@角色A为准，后续动作重新生成。
```

错误：

```text
参考图、参考图2、参考图3都作为整体参考。
```

问题：多张参考图任务不清，模型不知道哪张图决定场景、造型、动作或风格。

更稳：

```text
参考图作为场景与光影参考，参考图2作为服装造型参考，参考图3作为动作站位参考，各参考图分工明确。
```

## 9. 平台素材调用逻辑

除参考图和 @角色外，用户也可能明确指定平台内素材，例如视频、音频、动作参考等。

这类素材调用要以用户命名为准。

### 9.1 动作视频参考

```text
视频参考作为动作节奏参考，保留转身、遮挡镜头和卡点节奏，但替换为原创场景与原创服装。
```

### 9.2 音频参考

```text
音频参考作为背景音乐节奏，角色动作与鼓点同步，结尾在重拍处定格。
```

### 9.3 产品图参考

```text
参考图中的产品作为核心物件，保持产品外观清晰稳定，镜头从产品细节特写拉开到使用场景。
```

### 9.4 多图素材参考

```text
参考图、参考图2、参考图3作为不同素材来源，分别承担场景、造型、动作、道具、光影或风格参考任务。Prompt 中必须逐一说明每张图的用途。
```

示例：

```text
参考图作为舞台背景参考；参考图2作为角色服装与配件参考；参考图3作为双人动作和站位参考。三张图只提供视觉参考，不改变@角色的人物身份。
```

### 9.5 素材调用原则

1. 用户如何命名素材，Prompt 就尽量沿用用户命名。
2. 不主动发明不存在的素材名称。
3. 每个素材都要写清用途。
4. 不同素材之间要分工明确。
5. 若素材用途不明确，优先根据用户目标推断一个主要用途。
6. 多张参考图应逐张说明用途，避免写成笼统「参考这些图」。
7. 若多张素材互相冲突，应明确主素材与辅助素材。

## 10. 数字人机制

### 10.1 数字人的用途

数字人适合：

* 用户本人出镜
* 固定角色系列
* 直播间模拟
* MV 主角
* 剧情短片男女主
* 广告代言式表达
* 口播 / 演讲 / 访谈 / 反应类视频

### 10.2 数字人 Prompt 写法

```text
@角色A作为主角，保持面部、发型与身份一致，穿简洁蓝色西装，站在TED演讲舞台中央，镜头从中景缓慢推进到近景，他自信看向观众，舞台灯光干净专业，背景大屏显示 “AI VIDEO CREATION”。
```

### 10.3 双人数字人写法

```text
@角色A站在画面左侧，@角色B站在画面右侧，两人面对镜头轮流对嘴演唱，动作同步但不重叠，镜头稳定中景，保持两人面部清晰、服装一致、无多余人物。
```

### 10.4 人物稳定建议

当用户使用 @角色时，GPT 应自动加入以下稳定约束：

```text
保持@角色面部一致，服装发型一致，主体始终清晰居中，动作连贯缓慢，避免面部变形，无多余人物，无穿模。
```

### 10.5 数字人高风险动作

以下动作容易导致脸部不稳或身体异常，应尽量弱化：

* 快速翻滚
* 高速打斗
* 连续转身
* 大幅度跳舞
* 多人拥挤互动
* 镜头快速绕脸
* 遮挡脸部后快速切换
* 强特效覆盖面部

更稳替代：

| 高风险动作  | 稳定替代        |
| ------ | ----------- |
| 快速打斗   | 慢动作蓄力、定格出招  |
| 疯狂跳舞   | 简洁手势 + 节奏点头 |
| 快速转圈   | 半身缓慢转身      |
| 多人贴脸互动 | 双人中景，明确左右位置 |
| 强光覆盖脸  | 轮廓光，不遮挡五官   |

## 11. 时长与节奏机制

### 11.1 默认 12 秒结构

即梦短视频默认结构：

```text
0–3秒：开场钩子
3–6秒：主体动作推进
6–10秒：高潮 / 反转 / 视觉变化
10–12秒：结尾记忆点 / 定格 / 台词落点
```

### 11.2 不同类型时长建议

| 类型   | 建议时长                      | 结构重点                  |
| ---- | ------------------------- | --------------------- |
| 搞笑反转 | 8–12 秒                    | 快速建立反差，结尾落包袱          |
| 变装卡点 | 8–12 秒                    | Before → 遮挡转场 → After |
| 剧情短片 | 12–30 秒（默认压缩到12秒，明确要求可延长） | 钩子 → 冲突 → 转折 → 余味     |
| 广告展示 | 8–12 秒                    | 场景痛点 → 产品出现 → 记忆点     |
| MV舞蹈 | 10–12 秒                   | 主歌/副歌节奏 + 舞蹈落点        |
| 视觉奇观 | 6–12 秒                    | 异常视觉 → 变化过程 → 高潮定格    |
| 口播演讲 | 10–30 秒                   | 观点开场 → 表达重点 → 强收尾     |

### 11.3 节奏原则

1. 前 2–3 秒必须让观众知道「为什么要看」。
2. 中段每 2–3 秒给一个新信息点。
3. 结尾必须有定格、台词、反转、视觉高潮或情绪余味。
4. 不要把背景解释放在开头。
5. 短视频优先「先给结果，再给过程」。

## 12. 比例与构图规则

### 12.1 竖屏 9:16

适合：

* 抖音 / TikTok / Reels / Shorts
* 人像出镜
* 直播间模拟
* 变装
* 情绪短片
* 搞笑反转

构图建议：

* 主体放中轴线或上半区。
* 脸部不要太靠近边缘。
* 下方预留字幕安全区。
* 多人画面要左右分明。
* 道具或文字不要挡住脸。

### 12.2 横屏 16:9

适合：

* 电影感场景
* 群像调度
* 宏大空间
* 武侠 / 战争 / 史诗感
* YouTube 横屏展示

### 12.3 1:1 或其他比例

适合：

* 图文封面
* 产品展示
* 社交平台方形内容

如果用户没有说明比例，默认使用 9:16。

## 13. 字幕与文字机制

### 13.1 文字生成风险

AI 视频模型生成文字时可能出现：

* 错字
* 模糊
* 抖动
* 乱码
* 位置漂移
* 被人物或道具遮挡
* 多帧不一致

因此默认建议：

```text
重要字幕、歌词、标题、Logo 建议后期添加。
```

### 13.2 什么时候可以让模型生成文字

可以尝试的情况：

* 文字很短
* 文字只出现一次
* 背景稳定
* 镜头移动很慢
* 字幕在画面固定区域
* 用户明确要求画面内出现文字

### 13.3 文字写法规则

如果用户要求画面出现文字，必须使用中文全角引号标记：

```text
画面底部中央出现清晰白色字幕 “I am sorry”
```

```text
舞台大屏显示 “AI VIDEO CREATION”
```

```text
直播间弹幕出现 “家人们，上链接！”
```

### 13.4 文字稳定写法

```text
文字固定在画面底部中央，白色清晰字体，背景干净，镜头稳定，无物体遮挡文字。
```

### 13.5 不建议模型直接生成的文字

* 长歌词
* 多行字幕
* 复杂菜单
* 品牌 Logo
* 密集弹幕
* 表格
* 小字号说明
* 复杂英文长句

处理建议：

```text
画面预留字幕安全区，字幕建议后期添加。
```

## 14. 拍同款机制

### 14.1 拍同款的正确理解

拍同款不是复制原视频，而是复用：

* 节奏结构
* 镜头逻辑
* 情绪曲线
* 转场方式
* 爆点位置
* 视觉风格方向

必须替换：

* 具体角色
* 具体台词
* 具体剧情细节
* 受版权保护的画面
* 品牌 / IP / Logo
* 原音乐和原素材

### 14.2 拍同款拆解模板

```text
【原视频结构】
开场钩子：
中段推进：
镜头节奏：
风格特征：
结尾记忆点：
可复用部分：
必须替换部分：

【原创改写方向】
主体：
场景：
动作：
镜头：
风格：
结尾：
```

### 14.3 拍同款输出原则

1. 先拆解，再改写。
2. 保留节奏，不保留具体表达。
3. 替换人物、场景、台词、道具。
4. 提醒使用原创素材。
5. 输出时用「原创改编版」而非「复刻」。

## 15. 首帧 / 尾帧机制

### 15.1 首帧用途

首帧适合：

* 锁定角色造型
* 锁定构图
* 锁定场景
* 锁定广告产品
* 锁定变装前状态
* 锁定封面感画面

参考图作为首帧时，不使用 @ 符号，直接写「参考图」。

写法：

```text
参考图作为首帧构图，镜头从该画面开始，主角缓慢抬头看向镜头。
```

如果同时使用 @分身：

```text
@角色A作为主角，参考图作为首帧构图与服装风格参考，镜头从参考图相似画面开始，主角缓慢抬头看向镜头。
```

如果使用多张参考图：

```text
参考图作为首帧构图参考，参考图2作为服装造型参考，参考图3作为动作站位参考。镜头从参考图相似构图开始，角色身份以@角色为准。
```

### 15.2 尾帧用途

尾帧适合：

* 变装后定格
* 产品最终展示
* 反转结尾
* 海报感收束
* 情绪落点

写法：

```text
结尾停在海报感定格画面，主角站在画面中央，灯光聚焦，表情坚定。
```

多参考图尾帧写法：

```text
参考图3作为尾帧情绪和构图参考，结尾停在接近参考图3的站位与画面关系，但人物身份仍以@角色为准。
```

### 15.3 首尾帧稳定原则

1. 首帧不要承担太多动作。
2. 尾帧要明确停在哪里。
3. 变装类视频必须说明遮挡或转场方式。
4. 如果需要封面头帧，可写「开头闪现一帧封面感画面」。
5. 多张参考图中，只有明确指定为首帧或尾帧的图片才承担首尾帧任务。

## 16. 音频与节奏机制

### 16.1 声音类型

| 声音   | 用途          |
| ---- | ----------- |
| BGM  | 控制整体情绪和节奏   |
| 环境音  | 增强真实感       |
| 同步音效 | 强化动作落点      |
| 台词   | 建立角色与笑点     |
| 旁白   | 解释剧情或增强情绪   |
| 歌词   | MV / 音乐内容核心 |

### 16.2 声音写法

```text
配轻快喜剧BGM，箭射中瞬间加入夸张“咻——砰”音效，结尾音乐突然停顿形成笑点。
```

```text
背景为低频悬疑氛围音乐，脚步声和呼吸声清晰，门打开瞬间音乐戛然而止。
```

```text
强节拍K-pop电子鼓点，动作与鼓点卡点，副歌处镜头快速切换。
```

### 16.3 音画同步原则

1. 动作高潮要和音效同步。
2. 卡点视频必须明确哪个动作对齐节拍。
3. 搞笑视频结尾适合音乐停顿。
4. 情绪视频不要让音乐压过台词。
5. 口播视频以人声清晰为第一优先。

## 17. 合规与版权边界

### 17.1 人像与授权

使用真实人物、他人照片、公众人物、明星、IP 角色时，必须注意授权与平台限制。

安全处理方式：

* 使用用户本人授权的 @数字人。
* 用原创角色替代公众人物。
* 用风格描述替代具体明星脸。
* 用「港片喜剧风」「武侠电影感」替代具体电影角色。
* 用「原创怪兽」替代已有影视怪兽。

### 17.2 品牌与 Logo

不建议直接生成真实品牌 Logo，除非用户明确拥有授权或用于内部合法场景。

更稳写法：

```text
无品牌Logo的高端运动饮料瓶
```

```text
原创虚构科技公司发布会舞台
```

### 17.3 IP 风格与致敬

可以描述风格方向：

```text
90年代港片喜剧质感
复古邵氏武侠电影氛围
好莱坞预告片式节奏
日系青春电影感
```

避免直接要求：

```text
完全复刻某电影名场面
使用某知名角色造型
照搬某影视台词
```

### 17.4 拍同款合规原则

```text
学习结构，替换表达。
学习风格，原创内容。
学习节奏，不复制素材。
```

## 18. 常见失败原因与修正

### 18.1 主体不稳定

原因：

* 人物太多
* 动作太快
* 风格词过强
* 脸部被遮挡
* 服装描述前后冲突
* 多张参考图没有分工
* 参考图人物身份与 @角色混淆

修正：

```text
减少人物数量，保持主角居中，中近景，动作缓慢，服装发型一致，面部清晰无遮挡。
```

多参考图修正：

```text
参考图负责场景，参考图2负责服装，参考图3负责动作站位；人物身份以@角色为准，不参考图片人脸。
```

### 18.2 动作混乱

原因：

* 同时发生太多动作
* 动作链太长
* 镜头变化太快
* 场景与动作不匹配
* 多张参考图中的动作要求互相冲突

修正：

```text
一镜一动作，动作分为开始、过渡、高潮、落点，减少同时发生的事件。
```

多参考图修正：

```text
只让参考图3承担动作站位任务，其他参考图不参与动作设计。
```

### 18.3 风格跑偏

原因：

* 风格关键词过多
* 同时要求写实、动漫、油画、赛博、国风
* 光色与场景冲突
* 多张参考图风格不一致但没有指定主风格

修正：

```text
保留一个主风格，例如16mm暖黄胶片，其他风格删除，只补充光影和质感。
```

多参考图修正：

```text
以参考图作为主视觉风格，参考图2和参考图3只提供造型与动作，不影响整体调色。
```

### 18.4 字幕乱码

原因：

* 文字太长
* 字体太小
* 镜头移动太快
* 背景复杂

修正：

```text
减少文字长度，固定底部中央，背景干净，镜头稳定；长字幕建议后期添加。
```

### 18.5 视频不够爆

原因：

* 开头没有钩子
* 中段没有变化
* 结尾没有记忆点
* 音效没有落点

修正：

```text
前2秒直接给异常画面或反差，中段每2-3秒增加一个信息点，结尾加入台词、定格或反转。
```

### 18.6 多参考图混乱

原因：

* 多张参考图都写成「整体参考」
* 没有指定参考图、参考图2、参考图3各自用途
* 多张参考图同时控制人物脸部、服装、动作和场景
* 参考图之间风格冲突但没有指定主图

修正：

```text
参考图作为主场景和光影参考；参考图2作为服装、发型和配件参考；参考图3作为动作站位和构图参考。人物身份以@角色为准，参考图不控制人物脸部。
```

## 20. 本库给 GPT 的执行指令

当调用本库时，GPT 应遵守：

1. 输出 Prompt 时优先使用 Canvas 作为可编辑版本；如无法使用 Canvas，或用户需要复制，再补充 Markdown code block。
2. 用户没说比例，默认 9:16。
3. 用户没说时长，默认 12 秒。
4. 用户有 @角色，必须强化人物稳定。
5. 用户有参考图，必须说明参考图用途，并直接使用「参考图」关键词，不要写成 @参考图。
6. 用户有多张参考图，必须按照「参考图、参考图2、参考图3……」逐张说明用途。
7. 多张参考图之间要分工明确，不要笼统写「参考这些图」。
8. 多张参考图与 @角色同时出现时，@角色负责人物身份，参考图负责视觉参考。
9. 用户要字幕，必须判断是否建议后期添加。
10. 用户要拍同款，必须拆结构，不复制表达。
11. 用户要电影感，必须补充镜头、光影、节奏，而不是堆形容词。
12. 用户要爆款，必须强化前 2 秒钩子和结尾记忆点。
13. 输出最终 Prompt 时，默认采用「双版本输出」：先给完整导演版，确保内容、镜头、情绪与结构完整；再在同一回覆中额外给 ≤580字压缩版。
14. 只对压缩版做字符数压缩；完整版不得因字数限制提前删减核心内容。
15. 压缩版优先保留主体、动作、镜头、风格、约束，并标注字符数；计算时包含中文、英文、数字、标点、空格与换行。
16. 输出最终 Prompt 前，必须做平台适配检查。
17. 用户提出图片素材相关需求时，必须先拆成最多三张素材参考图需求：纯背景图、造型展示图、动作与运镜草图。

### 21.3 @数字人稳定模板

```text
@角色作为主角，保持面部、发型、服装一致，主体始终清晰居中，动作缓慢连贯，镜头稳定，无多余人物，无穿模，无面部变形。
```


### 21.5 拍同款安全模板

```text
参考原视频的节奏结构与镜头逻辑，但替换为原创角色、原创场景、原创台词和原创动作，不复制原画面、不使用原品牌/IP/音乐。
```

### 21.6 字幕安全模板

```text
画面预留底部字幕安全区，重要字幕建议后期添加；如需画面内文字，仅显示短句 “{文字内容}”，白色清晰字体，固定底部中央。
```

### 21.7 12秒结构模板

```text
0–3秒：异常画面或强反差钩子。
3–6秒：主体动作推进，观众理解场景和人物。
6–10秒：冲突升级、视觉变化或笑点爆发。
10–12秒：定格、台词、反转或海报感记忆点。
```

## 22. 总结

本库的核心价值是让即梦 Prompt 更符合平台生成逻辑。

最重要的判断句：

```text
即梦不是读关键词，而是在理解一个可视化动作场景。
```

因此，每次生成都要确保：

```text
主体清楚，动作连续，镜头明确，风格统一，时长合理，约束稳定。
```

如果存在多张参考图，还要确保：

```text
每张参考图用途清楚，主次明确，彼此不冲突；@角色负责人脸身份，参考图负责视觉参考。
```
